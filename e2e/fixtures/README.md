# Visual-verify fixtures

These fixtures back the harness in `e2e/visual-verify/`. Each subdir
holds a `.kicad_mod` whose `(model …)` line points at a real `.step`
file in the same directory (paths use `${KSL_ROOT}/UFL_KSL/UFL_KSL.3dshapes/<name>`
so they resolve inside the smoke-ui workspace at `/tmp/e2e-workspace/`).

## Setup

```bash
bash e2e/fixtures/install-deps.sh         # ~80 MB venv with cadquery
source e2e/fixtures/.venv/bin/activate
python3 e2e/fixtures/synthetic_pcb_named/gen_step.py   # regen synthetic STEP
```

## Fixtures

### `u_fl_hirose/` — IPEX surrogate

* **`U.FL_Hirose_U.FL-R-SMT-1_Vertical.kicad_mod`** — copied from KiCad 9
  stock library inside the local containerd snapshot
  `/var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/snapshots/1047/fs/usr/share/kicad/footprints/Connector_Coaxial.pretty/`.
  The original `(model …)` URL was rewritten to `${KSL_ROOT}/UFL_KSL/…`.
* **`U.FL_Hirose_U.FL-R-SMT-1_Vertical.step`** — downloaded from
  <https://gitlab.com/kicad/libraries/kicad-packages3D/-/raw/master/Connector_Coaxial.3dshapes/U.FL_Hirose_U.FL-R-SMT-1_Vertical.step>
  (free, public, no login). 70 KiB ISO-10303-21 STEP.
* **Intended bug exposure**: the user's original report is about IPEX
  20952-024E-02 (paywalled SnapEDA). U.FL is mechanically and
  topologically identical (micro-coax, ~3×3 mm body, dome housing). It
  reproduces:
  * Bug 3 — STEP→GLB load chain end-to-end on a tiny multi-mesh
    connector body.
  * Bug 1 — alignment: SnapEDA STEPs typically have non-zero local
    origin; this file's origin is the housing centroid.
* **Expected runtime values** (to be confirmed by harness on first run):
  * substrate mesh name: should match `preview_PCB` (the synthetic
    workspace board) — *not* anything inside the U.FL STEP itself.
  * chip node count: ~5–9 (housing shell + central pin + base ring +
    side flanges, depending on STEP tessellation).

### `usb_c_hro/` — large connector body

* **`USB_C_Receptacle_HRO_TYPE-C-31-M-12.kicad_mod`** — copied from
  `…/snapshots/1049/fs/usr/share/kicad/demos/royalblue54L_feather/lib/footprints/RoyalBlue54L-Feather-Connector_USB.pretty/`.
  Stripped the `.wrl` `(model …)` block; rewrote the `.STEP` block to
  `${KSL_ROOT}/UFL_KSL/UFL_KSL.3dshapes/USB_C_Receptacle_HRO_TYPE-C-31-M-12.STEP`.
* **`USB_C_Receptacle_HRO_TYPE-C-31-M-12.STEP`** — copied from the same
  demo's `lib/3dmodels/Connector_USB.3dshapes/`. 698 KiB, ISO-10303-21.
* **Intended bug exposure**: USB-C body is ≈ 9×7×3 mm — *bigger* than
  the synthetic substrate slabs we generate, so the
  "largest-XZ-area-wins" fallback in `findSubstrateMesh` would
  mis-pick the connector if the exact-name match misfires (Bug B).
* **Expected runtime values**:
  * substrate mesh name: must remain `preview_PCB`.
  * chip node count: 10+ (shell, contacts, plastic core, mounting tabs).

### `synthetic_pcb_named/` — Bug A in CI

* **`gen_step.py`** — CadQuery + OCCT XCAF script. Builds a STEP with
  two top-level labelled solids written via `STEPCAFControl_Writer` (so
  kicad-cli's `RWGltf_CafWriter` keeps both meshes through GLB export):
  * `housing_PCB_BODY` (8×8×4 mm) — intentionally embeds "PCB" in
    the name to trip a naive `/pcb/i` substrate heuristic.
  * `connector_substrate_PLATE` (6×6×0.6 mm slab) — a connector base
    plate. **Deliberately NOT named `preview_PCB`**: kicad-cli already
    auto-generates a board substrate mesh called `preview_PCB`, so a
    fixture solid sharing that name would produce a duplicate-named
    node in the GLB and confuse `findSubstrateMesh`.
* **`synthetic_pcb_named.step`** — generated output, ~33 KiB.
* **`synthetic_pcb_named.kicad_mod`** — minimal hand-rolled footprint
  with two SMD pads. The `(model …)` line points at the `.step` file
  by basename — solid renames inside the STEP do not affect this line.
* **Intended bug exposure**: hostile mesh names. A naive
  `name.match(/pcb/i)` heuristic returns the *chip* mesh first (because
  it's enumerated first in the GLB). The fixed heuristic must prefer
  the exact-match `preview_PCB` node generated by kicad-cli.
* **Expected runtime values** (after kicad-cli GLB export):
  * GLB node names: `housing_PCB_BODY`, `connector_substrate_PLATE`,
    plus kicad-cli's auto-generated `preview_PCB` board node.
  * substrate mesh name: `preview_PCB` (the auto board, uniquely).
  * chip node count: 2 (`housing_PCB_BODY` + the connector plate)
    once the substrate election succeeds.

## Verification commands

```bash
ls -la e2e/fixtures/u_fl_hirose/ e2e/fixtures/usb_c_hro/ e2e/fixtures/synthetic_pcb_named/
e2e/fixtures/.venv/bin/python3 -c "import cadquery; print(cadquery.__version__)"
e2e/fixtures/.venv/bin/python3 -c "
from OCP.STEPControl import STEPControl_Reader
for p in [
  'e2e/fixtures/u_fl_hirose/U.FL_Hirose_U.FL-R-SMT-1_Vertical.step',
  'e2e/fixtures/usb_c_hro/USB_C_Receptacle_HRO_TYPE-C-31-M-12.STEP',
  'e2e/fixtures/synthetic_pcb_named/synthetic_pcb_named.step',
]:
  r = STEPControl_Reader(); r.ReadFile(p); r.TransferRoots()
  print(p, 'shape_null=', r.OneShape().IsNull())
"
```

## Quirks / known gaps

* **No real IPEX 20952-024E-02 STEP.** SnapEDA paywalls it. If you
  obtain the file manually, drop it next to U.FL in
  `e2e/fixtures/ipex_20952/` and add a matching footprint — the harness
  will pick it up if the directory follows the same convention.
* **`.wrl` model line removed from USB-C kicad_mod.** The original
  KiCad demo lists both a WRL and a STEP model. We only ship the STEP
  to keep the fixture self-contained.
* **`numpy<2` pin in `install-deps.sh`.** cadquery 2.4.0 transitively
  depends on `nptyping 2.0.1`, which references `np.bool8` — removed in
  numpy 2.x. The pin is documented in the script.
* **`file(1)` reports STEP files as "ASCII text".** That is correct;
  ISO-10303-21 STEP is plain text starting with `ISO-10303-21;`. Use
  `head -c 14 file.step | grep -q "ISO-10303-21"` for a binary-safe
  identification, not `file`.
