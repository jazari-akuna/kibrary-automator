# Kibrary Automator — P2 (Library Management) + cross-cutting extras

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Ship the Library Management room (browse / edit / move / rename / delete committed parts) AND four cross-cutting features the user pulled forward from P3: sidecar auto-detect+install, external STEP file browse/replace, GitHub-releases auto-update, and signed installers per platform.

**Architecture:** Same Tauri + SolidJS + Python sidecar stack from P1. New "Libraries" room composes existing preview/editor blocks reused from the staging flow. Sidecar bootstrap lives in Rust and runs before the main window opens. Auto-update via `tauri-plugin-updater` against GitHub Releases JSON endpoint. Signing configured per-OS at `cargo tauri build` time.

**Spec reference:** `docs/superpowers/specs/2026-04-25-kibrary-redesign.md` §10 (Library management) + §14 (open questions). The new extras are a scope-pull-forward from §11 (P3).

**Sub-phases:**
- **2A — Library room** (P1–P9): browse/edit/move/rename/delete + library metadata editor + diff preview
- **2B — Sidecar bootstrap** (P10–P12): detect-or-install python3 + kibrary_sidecar package
- **2C — External STEP** (P13–P14): browse + replace 3D model from disk
- **2D — Auto-update** (P15–P17): tauri-plugin-updater + GitHub Releases endpoint + UI
- **2E — Signing** (P18–P21): apply the research recommendation per platform + CI/release docs

---

## Conventions for every task

- Work in `/root/kibrary-automator`, branch `p2`. Manager creates the branch off `main` (after `p1-mvp` merges) before dispatching workers.
- TDD where the task involves logic. Tests first → fail → implement → pass.
- **Workers do not commit; do not modify methods.py or registry.ts.** Manager wires those at end of each wave.
- After UI tasks: `pnpm screenshot /` and confirm visually before declaring done.

## Phase 2B — Sidecar bootstrap

### Task P10: Sidecar detection in Rust

**Files:**
- Create: `src-tauri/src/bootstrap.rs`, modify `src-tauri/src/main.rs`

Behavior:
1. On app start (before window opens), check: is `python3` available? Is `kibrary_sidecar` importable in that python?
2. If not: open a small bootstrap window asking the user how to install (auto-install via pip in a venv, or pick custom path, or quit).
3. Auto-install: shells out to `python3 -m venv ~/.local/share/kibrary/venv && ~/.local/share/kibrary/venv/bin/pip install kibrary-sidecar` (we'd publish to PyPI, or for now bundle a wheel inside the app and `pip install <bundled-wheel>`).
4. After install succeeds: spawn sidecar from the resolved python; proceed to main window.

- [ ] Tauri command `bootstrap.detect()` returns `{python_path, sidecar_present, version?}`.
- [ ] Tauri command `bootstrap.install({method, custom_path?})` runs the install, streams progress events.
- [ ] Bootstrap UI: a tiny separate Tauri window (`bootstrap.html` + `bootstrap.tsx` entry) that runs only this flow.
- [ ] Detection cached in `~/.config/kibrary/python.json`.
